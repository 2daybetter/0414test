# 예다함 홈페이지 운영·유지보수 및 온라인 광고 통합 운영 에이전트 시스템 설계서

> 목적: Claude Code 구현 참조용 계획서

## 1. 작업 컨텍스트

### 배경 및 목적
더케이예다함(yedaham.co.kr) 기 구축 시스템(2025-12-18 오픈)의 운영·유지보수 및 온라인 광고 통합 운영 사업이다. 계약기간 2026.03~2026.12.31(10개월), 총 사업비 427,920,000원. React19 + SpringBoot3.5.3 + Kotlin2.2.0 + Oracle 11g R2 기반의 기존 시스템을 인수하여 무중단 운영 체계를 구축하고, 24/365 장애 대응·월별 보고·온라인 광고 통합 운영을 수행한다.

### 범위
- 포함:
  - 홈페이지(yedaham.co.kr) 및 모바일 앱(React Native) 운영·유지보수
  - 운영자 시스템(CMS) 7개 모듈 콘텐츠 관리 대행
  - 온라인 광고 통합 운영 (네이버·구글·SNS·유튜브)
  - GA4 기반 월별 성과 보고서 생성
  - 결함 수정·보안 패치·기능 개선 (경미한 범위)
  - 24/365 모니터링 및 장애 대응 (4h 조치/24h 복구)
- 제외:
  - 신규 시스템 개발 (기 구축 시스템 운영만)
  - 광고비 실제 집행 (별도 예산, 계약 외)
  - 물리 인프라 구축 (KT Cloud CDN 기 운영)

### 입출력 정의

| 항목 | 내용 |
|------|------|
| **입력** | 기 구축 소스코드·DB, 운영자 시스템 접근 권한, GA4 데이터, 광고 계정 접근 권한, 클라이언트 요청사항 |
| **출력** | 월별 운영 보고서, 장애 처리 보고서, 보안 점검 결과, 광고 성과 리포트, 서비스 개선 결과물 |
| **트리거** | 클라이언트 요청 / 모니터링 경보 / 월별 정기 보고 일정 |

### 제약조건
- 기존 스택 유지 (React19 + SpringBoot3.5.3 + Kotlin2.2.0 + Oracle 11g R2) — 임의 기술 교체 금지
- 배포 전 스테이징 환경 검증 필수
- 개인정보 처리: SHA512+ 암호화 유지, 개인정보보호법 준수
- 장애 SLA: 인지 4시간 내 조치, 24시간 내 복구
- 월별 정기 보고서 제출 (PMR-001)
- 보안 취약점 점검 연 1회 이상 (SER-003)
- 비상주 투입 (필요 시 현장 방문 협의)

## 2. 워크플로우 정의

### 전체 흐름도

```
[킥오프 및 인수인계]
        ↓
[Step 1: 시스템 인수인계 및 환경 파악]
        ↓
[Step 2: 모니터링·장애 대응 체계 구축]
        ↓
[Step 3: 정기 운영 사이클 (월 반복)]
     ┌──┴──┐
     │     │
[결함/기능 처리] [광고 운영]
     └──┬──┘
        ↓
[Step 4: 월별 보고서 생성·제출]
        ↓
[다음 달 사이클 반복 또는 종료 (2026.12.31)]
```

### LLM 판단 vs 코드 처리 구분

| LLM이 직접 수행 | 스크립트로 처리 |
|----------------|----------------|
| 결함 우선순위 분류 (심각도·영향도 평가) | GA4 Reporting API 데이터 수집 (`scripts/ga4_report.py`) |
| 광고 성과 분석 및 개선 방향 도출 | 보안 취약점 자동 스캔 (`scripts/security_scan.sh`) |
| 요구사항 공수 추정 및 일정 협의 | 모니터링 경보 수신 및 알림 (`scripts/monitor_alert.py`) |
| 월별 운영 보고서 내러티브 작성 | 백업 상태 확인 (`scripts/backup_check.sh`) |
| 위험 요소 식별·평가 | 빌드·배포 실행 (`scripts/deploy.sh`) |
| 광고 소재·콘텐츠 방향 결정 | 성능 지표 수집 (응답시간·에러율) |

### 단계별 상세

#### Step 1: 시스템 인수인계 및 환경 파악

- **처리 주체**: 에이전트 + 스크립트
- **입력**: 전임 운영사 인수인계 문서, 소스코드 레포지토리 접근 권한, 서버 접속 정보
- **처리 내용**:
  - 기존 소스코드 구조 파악 (React19 프론트, SpringBoot/Kotlin 백엔드)
  - DB 스키마·데이터 현황 파악 (Oracle 11g R2)
  - 배포 파이프라인 파악 및 문서화 (CI/CD 도구·브랜치 전략)
  - CMS 7개 모듈 접근 권한 및 운영 현황 파악
  - 광고 계정(네이버·구글·SNS) 접근 권한 인수
  - GA4 계정 접근 및 기존 데이터 현황 파악
  - 운영 매뉴얼 초안 작성
- **출력**: `docs/system-overview.md`, `docs/deploy-guide.md`, `docs/cms-guide.md`, `docs/ad-account-guide.md`
- **성공 기준**: 4개 문서 작성 완료, 배포 1회 성공적 실행, CMS 전체 모듈 접근 확인
- **검증 방법**: 규칙 기반 (문서 필수 항목 완성 여부) + 사람 검토 (클라이언트 확인)
- **실패 시 처리**: 인수인계 미흡 항목 에스컬레이션 (클라이언트에 전임사 협조 요청)

#### Step 2: 모니터링·장애 대응 체계 구축

- **처리 주체**: 에이전트 + 스크립트
- **입력**: Step 1 시스템 현황 문서
- **처리 내용**:
  - 24/365 모니터링 설정 (서버 CPU·메모리·응답시간·에러율 임계값 설정)
  - 장애 등급 정의 (Critical/Major/Minor) 및 대응 절차 수립
  - 알림 채널 설정 (이메일·Slack·SMS)
  - 백업 정책 수립 및 자동화 (DB 일일 백업, 파일 주별 백업)
  - 장애 보고서 템플릿 작성
  - 스테이징 환경 확인 및 배포 프로세스 표준화
- **출력**: `docs/monitoring-guide.md`, `docs/incident-response.md`, `scripts/monitor_alert.py`, `scripts/backup_check.sh`
- **성공 기준**: 모니터링 경보 테스트 성공, 장애 대응 절차 문서 완성, 백업 1회 실행 성공
- **검증 방법**: 규칙 기반 (경보 테스트 통과) + 사람 검토
- **실패 시 처리**: 모니터링 도구 접근 불가 시 클라이언트 에스컬레이션

#### Step 3: 정기 운영 사이클 (월 반복)

- **처리 주체**: 에이전트
- **입력**: 클라이언트 요청사항, 모니터링 경보, 전월 보고서
- **처리 내용**:

  **결함·기능 처리 (MPR-002, MPR-009)**
  - 클라이언트 요청 접수 → 심각도 분류 (Critical/Major/Minor)
  - 수정 범위 추정 및 일정 협의
  - 스테이징 환경 개발·검증 → 프로덕션 배포
  - 처리 결과 클라이언트 확인

  **CMS 콘텐츠 운영 (MPR-001)**
  - 콘텐츠 갱신 요청 처리 (상품 정보·공지·이미지 등)
  - 운영자 시스템 7개 모듈 운영 지원

  **온라인 광고 운영**
  - 광고 채널별 성과 주간 모니터링 (CTR·CPC·전환율)
  - 저성과 키워드·소재 교체 의사결정
  - SNS 월별 콘텐츠 캘린더 실행
  - 타깃 분석 및 캠페인 최적화 제안

  **보안·성능 유지 (SER, PER-001)**
  - 월별 보안 패치 확인 및 적용 (SpringBoot·Node.js 의존성 업데이트)
  - 응답 속도 3초 이내 유지 확인
  - 접근 권한 월별 점검

- **출력**: 처리된 결함/요청 목록, 배포 이력, 광고 주간 성과 데이터, 콘텐츠 갱신 이력
- **성공 기준**: 접수 요청 100% 응답, 장애 SLA 준수, 광고 성과 데이터 수집 완료
- **검증 방법**: 규칙 기반 (SLA 준수율, 요청 처리율) + LLM 자기 검증 (보고서 품질)
- **실패 시 처리**: SLA 미준수 시 즉시 에스컬레이션; 광고 계정 접근 불가 시 백업 계정 사용

#### Step 4: 월별 보고서 생성·제출 (PMR-001, PSR-001)

- **처리 주체**: 에이전트 + 스크립트 (`scripts/ga4_report.py`)
- **입력**: GA4 Reporting API 데이터, 결함 처리 이력, 광고 성과 데이터, 전월 보고서
- **처리 내용**:
  - GA4 API 호출 → 채널별 유입·세션·전환·이탈률 데이터 수집
  - 광고 채널별 성과 취합 (CTR·CPC·ROAS·전환수)
  - 결함 처리 현황 (접수·완료·보류 건수, SLA 준수율)
  - 보안·성능 지표 (응답 시간 평균·장애 발생 건수·취약점 현황)
  - 다음 달 운영 계획 및 위험 요소
  - 보고서 문서 생성 (Markdown → PDF)
- **출력**: `output/YYYY-MM-monthly-report.md`, 클라이언트 제출용 PDF
- **성공 기준**: 보고서 내 모든 지표 항목 데이터 포함, 월말 기준일 내 제출
- **검증 방법**: 규칙 기반 (필수 섹션 존재 확인) + LLM 자기 검증 (내러티브 품질·정확성)
- **실패 시 처리**: GA4 API 데이터 지연 시 수동 데이터로 대체 + 지연 사유 명시

### CLAUDE.md 핵심 섹션 목록

- 프로젝트 개요: 10개월 운영·유지보수 계약, 기존 시스템 스택 및 구조 요약
- SLA 기준: 장애 4h 조치/24h 복구, 응답 3초 이내 (PER-001)
- 요구사항 분류표: MPR/MHR/PER/DAR/SER/COR/PSR/PMR 전체 목록 및 처리 기준
- 배포 프로세스: 스테이징 검증 → 프로덕션 배포 단계별 절차
- 보안 원칙: 개인정보 SHA512+ 암호화, 최소 권한 원칙, 취약점 발견 시 48h 패치
- 보고서 주기: 월별 보고서 필수 항목 및 제출 기준일

### 에이전트 구조

**구조 선택**: 단일 에이전트 (메인) + 2개 서브에이전트

**선택 근거**: 운영 사이클은 메인 에이전트가 오케스트레이션하되, GA4 보고서 생성과 장애 대응은 독립적인 컨텍스트와 도구가 필요하므로 서브에이전트로 분리. 두 서브에이전트는 서로 의존하지 않음.

#### 메인 에이전트 (CLAUDE.md)
- **역할**: 월별 운영 사이클 전체 오케스트레이션, 요청 접수·분류·일정 관리
- **담당 단계**: Step 1, Step 2, Step 3 (결함·CMS·광고 운영)

#### 서브에이전트 목록

| 이름 | 역할 | 트리거 조건 | 입력 | 출력 | 참조 스킬 |
|------|------|-----------|------|------|----------|
| `monthly-reporter` | GA4 데이터 수집·보고서 생성 | 매월 말일 또는 수동 호출 | GA4 계정 정보, 처리 이력 | `output/YYYY-MM-monthly-report.md` | `monthly-reporter` |
| `incident-handler` | 장애 감지·대응·보고서 작성 | 모니터링 경보 수신 시 | 경보 내용, 시스템 접속 정보 | 장애 처리 보고서, 복구 이력 | `incident-handler` |

### 스킬 생성 규칙

> 이 설계서에 정의된 모든 스킬은 구현 시 반드시 `skill-creator` 스킬(`/skill-creator`)을 사용하여 생성할 것.
> 직접 SKILL.md를 수동 작성하지 말 것 — 규격 불일치 및 트리거 실패의 원인이 됨.

skill-creator가 보장하는 규격:
1. SKILL.md frontmatter (`name`, `description`) 필수 필드 준수
2. `description`의 트리거 정확도 최적화 (eval 기반 optimization loop)
3. 폴더 구조 (`SKILL.md` + `scripts/` + `references/`) 규격 준수
4. Progressive disclosure: SKILL.md 본문 500줄 이내, 대용량 참조는 `references/`로 분리
5. 테스트 프롬프트 실행 및 품질 검증 완료
