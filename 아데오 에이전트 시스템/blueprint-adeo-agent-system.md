# 아데오 그룹 통합 에이전트 시스템 설계서

> 작성일: 2026-04-16
> 목적: Claude Code 구현 참조용 계획서

---

## 1. 작업 컨텍스트

### 배경 및 목적

아데오 그룹은 비서실 + 4개 자회사(제안사·구축사·운영사·연구소), 총 17개 팀으로 구성된 웹 생태계 전문 그룹이다.
현재 자회사별 반복 업무(제안서 작성, 프로젝트 킥오프 문서, IA·화면설계서 작성 등)가 수작업으로 처리되고 있으며,
자회사 간 문서 포맷이 상이하여 협업 효율이 저하되고 있다.

이 에이전트 시스템이 해결하는 문제:
1. **반복 업무 자동화** — 제안사·구축사 핵심 업무를 에이전트가 자율 처리
2. **산출물 표준화** — 모든 산출물이 아데오 표준 문서 스타일로 통일
3. **지휘·모니터링 단일화** — 비서실이 4개 자회사를 단일 창구로 조율

### 범위

- **포함**: 비서실 L1 오케스트레이터, 4개 자회사 L2 에이전트, 17개 팀 L3 에이전트, 공통 스킬 라이브러리(8종), 표준 문서 템플릿(7종), 자회사 간 인계 워크플로우
- **제외**: 실제 웹사이트 코드 자동 생성, 외부 광고 플랫폼 API 연동(1차 범위 외), 재무·회계 시스템 연동, Figma 디자인 직접 출력

### 입출력 정의

| 항목 | 내용 |
|------|------|
| **입력** | 이사님/비서실: 자연어 지시 (예: "A사 홈페이지 제안서 만들어줘"). 팀원: 팀별 업무 요청 양식 또는 자연어 |
| **출력** | 아데오 표준 포맷 `.md` 문서. `/output/{자회사}/{팀}/{문서유형}-{날짜 또는 프로젝트명}.md` 경로 저장 |
| **트리거** | ① 이사님/팀원 직접 지시 ② 자회사 간 인계 이벤트 ③ 주간/월간 KPI 스케줄 트리거 |

### 제약조건

- **에이전트 계층 규칙**: 서브에이전트는 동급/하위 에이전트를 직접 호출 불가 — 반드시 상위 에이전트 경유
- **문서 표준**: 모든 산출물은 `/templates/` 내 해당 템플릿 참조 필수, 임의 포맷 금지
- **산출물 경로**: `/output/` 외 경로에 저장 금지
- **에이전트 계층**: 비서실(L1) → 자회사(L2) → 팀(L3), 역방향 지시 불가
- **재시도 한도**: LLM 자기 검증 실패 시 최대 2회 자동 재시도 후 에스컬레이션

### 용어 정의

| 용어 | 정의 |
|------|------|
| L1 에이전트 | 비서실 — 그룹 전체 오케스트레이터 (CLAUDE.md) |
| L2 에이전트 | 자회사 에이전트 — 자회사 내 팀 조율 (AGENT.md) |
| L3 에이전트 | 팀 에이전트 — 실무 단위 처리 (AGENT.md) |
| 표준 템플릿 | `/templates/` 에 저장된 아데오 문서 포맷 기준 파일 |
| 인계 이벤트 | 한 자회사에서 다음 자회사로 프로젝트가 이전되는 시점 |
| 산출물 패키지 | 하나의 워크플로우에서 생성된 모든 산출물의 집합 |

## 2. 워크플로우 정의

### 전체 흐름도

```
[이사님 / 팀원 지시 또는 스케줄 트리거]
                    ↓
         [L1: 비서실 에이전트]
          ↙    ↙    ↘    ↘
   [제안사] [구축사] [운영사] [연구소]
     ↓         ↓        ↓        ↓
  [팀 L3]   [팀 L3]  [팀 L3]  [팀 L3]
     ↓         ↓        ↓        ↓
  [표준 산출물 → /output/{자회사}/{팀}/]
                    ↑
     [비서실 수집 → KPI 대시보드 / 보고서]
                    ↓
              [이사님 보고]
```

### LLM 판단 vs 코드 처리 구분

| LLM이 직접 수행 | 스크립트로 처리 |
|----------------|----------------|
| 제안서 초안 작성 및 스토리보딩 | 파일 저장/로드 및 경로 관리 |
| 시장 트렌드·경쟁사 분석 | 템플릿 변수 치환 및 섹션 조립 |
| IA(정보구조) 설계 — FO/BO 계층 설계 | /output/ 디렉토리 스캔 및 KPI 집계 |
| 화면설계서 텍스트 명세 작성 | 산출물 필수 섹션 존재 여부 검증 (validate-doc.py) |
| 경쟁 PT 스토리보딩 | 스케줄 트리거 실행 |
| 정책자금 매칭 적합성 판단 | 자회사 간 인계 이벤트 파일 생성 |
| KPI 해석 및 보고서 서술 | 산출물 네이밍 규칙 적용 |
| 기술 리서치 요약 / DB 스키마 초안 작성 | 폴더 구조 자동 생성 |
| WBS 단계별 일정 추론 및 리스크 판단 | WBS 일정 계산 (가중치 기반) |
| 요구사항 우선순위 분류 | TC ID 자동 채번 및 집계 |
| API 엔드포인트 명세 작성 | 산출물 코드(PM-01~OP-05) 매핑 검증 |

### 핵심 워크플로우 A: 제안사 수주 프로세스 (최우선)

```
[수주 기회 입력 — 이사님 또는 영업팀]
               ↓
[Step A-1: 영업팀 — 기회 검토 및 분류]
               ↓
      [분기: 정책자금 해당?]
      ↙                   ↘
[Step A-2a: 정책자금팀]   [Step A-2b: 전략팀]
[정책자금 조회/매칭]      [시장분석/경쟁사 조사]
      ↘                   ↙
[Step A-3: 제안팀 — 제안서 초안 작성]
               ↓
[Step A-4: 제안팀 — 경쟁 PT 자료 제작]
               ↓
[Step A-5: 비서실 — 이사님 최종 검토 요청]
               ↓
[산출물: 제안서 패키지 → /output/제안사/]
```

#### Step 1: [A-1] 영업팀 — 수주 기회 검토

- **처리 주체**: 영업팀 L3 에이전트
- **입력**: 고객사명, 프로젝트 개요, 예산 규모, 납기, 담당자 정보
- **처리 내용**: 수주 기회 타당성 검토, 정책자금 해당 여부 판단(업종·규모 기준), 우선순위 분류(A/B/C)
- **출력**: `/output/제안사/영업팀/opportunity-{YYYYMMDD}.md`
- **성공 기준**: 고객사, 예산, 분류(정책자금/일반), 우선순위 등급이 모두 포함된 기회 검토서 생성
- **검증 방법**: 스키마 검증 — 고객사명·예산·분류·우선순위·납기 5개 필드 존재 확인
- **실패 시 처리**: 필수 정보 누락 시 사용자에게 재입력 요청 (최대 2회), 이후 에스컬레이션

#### Step 2: [A-2a] 정책자금팀 — 정책자금 조회 및 매칭

- **처리 주체**: 정책자금팀 L3 에이전트 + `policy-fund-finder` 스킬
- **입력**: 기회 검토서 (Step A-1 출력)
- **처리 내용**: 고객사 업종·규모·프로젝트 성격 기반 정부 지원사업 조회, 신청 조건 매칭, 사업제안서 초안 작성
- **출력**: `/output/제안사/정책자금팀/policy-fund-{YYYYMMDD}.md`
- **성공 기준**: 매칭 정책자금 목록, 신청 가능 여부, 예상 지원금 규모, 신청 마감일 포함
- **검증 방법**: 규칙 기반 — 매칭 결과 최소 1건 이상 또는 "해당 없음" 명시, 신청 마감일 필드 존재
- **실패 시 처리**: 매칭 결과 0건이면 "해당 없음" 리포트 생성 후 Step A-2b와 병합하여 계속 진행

#### Step 3: [A-2b] 전략팀 — 시장분석 및 경쟁사 조사

- **처리 주체**: 전략팀 L3 에이전트 + `strategy-analyzer` 스킬
- **입력**: 기회 검토서 (Step A-1 출력)
- **처리 내용**: 해당 업종 시장 트렌드 분석, 경쟁사 레퍼런스 조사, 차별화 포인트 도출
- **출력**: `/output/제안사/전략팀/market-analysis-{YYYYMMDD}.md`
- **성공 기준**: 시장 트렌드 요약, 경쟁사 3곳 이상 분석, 아데오 차별화 포인트 최소 3개
- **검증 방법**: 규칙 기반 — 경쟁사 섹션 3건 이상, 차별화 포인트 섹션 존재
- **실패 시 처리**: 자동 재시도 1회, 이후 전략팀 담당자 검토 에스컬레이션

#### Step 4: [A-3] 제안팀 — 제안서 초안 작성

- **처리 주체**: 제안팀 L3 에이전트 + `proposal-writer` 스킬
- **입력**: 기회 검토서 + 시장분석 리포트 + (선택) 정책자금 리포트
- **처리 내용**: 아데오 표준 제안서 템플릿(`/templates/proposal-template.md`) 기반 초안 작성, 고객사 맞춤 내용 삽입
- **출력**: `/output/제안사/제안팀/proposal-draft-{YYYYMMDD}.md`
- **성공 기준**: 표준 템플릿의 모든 필수 섹션 완성, 고객사명·프로젝트명·예산·일정 반영
- **검증 방법**: LLM 자기 검증 — 템플릿 섹션 완성도 및 고객사 정보 반영 여부 자체 평가
- **실패 시 처리**: 자동 재시도 최대 2회, 이후 제안팀 담당자 검토 에스컬레이션

#### Step 5: [A-4] 제안팀 — 경쟁 PT 자료 제작

- **처리 주체**: 제안팀 L3 에이전트 + `proposal-writer` 스킬 (PT 모드)
- **입력**: 제안서 초안 (Step A-3 출력)
- **처리 내용**: 제안서 핵심 메시지 추출, PT 슬라이드 구성 스크립트 작성 (텍스트 기반 스토리보드)
- **출력**: `/output/제안사/제안팀/proposal-pt-{YYYYMMDD}.md`
- **성공 기준**: 슬라이드별 제목·핵심 메시지·시각화 가이드 포함, 10슬라이드 이상
- **검증 방법**: 규칙 기반 — 슬라이드 섹션 10개 이상, 각 슬라이드에 제목·메시지·시각화 가이드 존재
- **실패 시 처리**: 자동 재시도 1회

#### Step 6: [A-5] 비서실 — 이사님 최종 검토 요청

- **처리 주체**: 비서실 L1 에이전트
- **입력**: 제안서 초안 + PT 자료 (Step A-3~4 출력)
- **처리 내용**: 산출물 패키지 취합, 검토 요약 보고서 생성, 이사님 승인 요청
- **출력**: `/output/비서실/review-request-{YYYYMMDD}.md`
- **성공 기준**: 검토 요약(프로젝트 개요, 제안 전략, 예상 수주금액) + 첨부 산출물 목록 포함
- **검증 방법**: 사람 검토 — 이사님 승인/반려 결정
- **실패 시 처리**: 반려 시 반려 사유와 함께 해당 단계(A-3 또는 A-4)로 재처리 지시

### 핵심 워크플로우 B: 구축사 프로젝트 실행 프로세스 (최우선)

아데오 표준 방법론 **PM(5%) → AY(10%) → DE(20%) → IM(45%) → TE(10%) → OP(10%)** 기반

```
[프로젝트 수주 확정 인계 — 제안사 → 구축사]
                ↓
[Step 7: PM 단계 — 착수 보고서 + WBS 작성]
                ↓
[Step 8: AY 단계 — 요구사항 정의 + 현황 분석]
                ↓
[Step 9: DE 단계-1 — IA 설계서 작성]
                ↓
[Step 10: DE 단계-2 — 화면설계서 + 디자인 스펙]
                ↓
[Step 11: DE 단계-3 — DB 설계 + 시스템 아키텍처 + API 명세]
                ↓
[Step 12: TE 단계 — 통합 테스트 시나리오 생성]
                ↓
[산출물: 프로젝트 문서 패키지 → /output/구축사/{프로젝트명}/]
```

#### Step 7: PM 단계 — 착수 보고서 + WBS 작성

- **처리 주체**: PM L3 에이전트 + `project-kickoff` 스킬
- **입력**: 제안사 인계 문서 (제안서, 계약 개요, 고객사 요구사항, 납기일)
- **처리 내용**: 착수 보고서(PM-01) 작성, 투입 인력 계획서(PM-02) 작성, WBS(PM-03) 초안 생성 — 6단계(PM/AY/DE/IM/TE/OP) 마일스톤 및 단계별 가중치(5/10/20/45/10/10%) 적용, 팀별 담당 업무·산출물 코드 배분, 리스크 목록 초안
- **출력**: `/output/구축사/{프로젝트명}/PM/kickoff-{프로젝트명}.md` (WBS + 착수보고서 통합)
- **성공 기준**: 6단계 마일스톤 모두 포함, 각 단계 시작/종료일·가중치·산출물 코드 명시, 팀별 담당 역할 정의, 납기일 반영
- **검증 방법**: 스키마 검증 — PM/AY/DE/IM/TE/OP 6개 단계 섹션 + 산출물 코드(PM-01~03) 존재 확인
- **실패 시 처리**: 자동 재시도 1회, 이후 PM 담당자에게 누락 항목 입력 요청

#### Step 8: AY 단계 — 요구사항 정의 + 현황 분석

- **처리 주체**: 웹기획팀 L3 에이전트 + `strategy-analyzer` 스킬
- **입력**: WBS(Step 7 출력) + 고객사 요구사항 + 벤치마킹 대상 URL
- **처리 내용**: 현황 분석서(AY-01) 작성 — 고객사 현황, 경쟁사 벤치마킹, 요구사항 정의(기능/비기능 요구사항 목록), 범위 확정, 서비스 컨셉 방향 정리
- **출력**: `/output/구축사/{프로젝트명}/웹기획팀/요구사항정의서-{프로젝트명}.md`
- **성공 기준**: 기능 요구사항 목록(최소 10개), 비기능 요구사항(성능/보안/반응형), 범위 확정(포함/제외), 컨셉 방향 포함

### CLAUDE.md 핵심 섹션 목록

- **비서실 역할 및 권한**: L1 오케스트레이터 정의, 자회사 에이전트 호출 권한 독점
- **자회사 에이전트 호출 경로**: 4개 L2 AGENT.md 파일 위치 및 호출 방법
- **KPI 모니터링 규칙**: 주간(매주 월요일)/월간(매월 1일) 자동 보고 기준
- **에스컬레이션 규칙**: 이사님 보고 기준 (재시도 초과, 고액 의사결정, 자회사 간 갈등)
- **표준 문서 경로**: `/templates/` 참조 의무화, 임의 포맷 금지 규칙
- **인계 이벤트 처리**: 자회사 간 프로젝트 인계 파일 형식 및 처리 절차

### 에이전트 구조

**구조 선택**: 멀티 에이전트 — L1 오케스트레이터(비서실) + L2 자회사 에이전트(4개) + L3 팀 에이전트(17개)

**선택 근거**: 5개 조직 × 17개 팀의 각기 다른 도메인 지식과 워크플로우를 단일 에이전트에 담으면 컨텍스트 초과 및 역할 혼선이 발생한다. L1이 전략 판단·자회사 조율을 담당하고, L2가 자회사 내 팀 조율, L3가 실무 처리를 분담하는 3계층 구조가 컨텍스트 효율과 역할 명확성 모두 최적화한다.

#### 메인 에이전트: 비서실 (CLAUDE.md)

- **담당 단계**: 워크플로우 C 전체, 워크플로우 A~E의 트리거 수신 및 최종 검토 단계

### 스킬 생성 규칙

> 이 설계서에 정의된 모든 스킬은 구현 시 반드시 `skill-creator` 스킬(`/skill-creator`)을 사용하여 생성할 것.
> 직접 SKILL.md를 수동 작성하지 말 것 — 규격 불일치 및 트리거 실패의 원인이 됨.

skill-creator가 보장하는 규격:
1. SKILL.md frontmatter (`name`, `description`) 필수 필드 준수
2. `description`의 트리거 정확도 최적화 (eval 기반 optimization loop)
3. 폴더 구조 (`SKILL.md` + `scripts/` + `references/`) 규격 준수
4. Progressive disclosure: SKILL.md 본문 500줄 이내, 대용량 참조는 `references/`로 분리
5. 테스트 프롬프트 실행 및 품질 검증 완료

### 4.2 AGENT.md 작성 규칙

모든 AGENT.md는 다음 **6개 섹션**을 완전히 정의해야 한다. 섹션 누락 또는 "TBD" 기재 금지.

```
## 역할 및 책임
## 트리거 조건
## 워크플로우 단계별 수행 지침
## 사용 스킬 목록
## 핵심 분기 확인 기준
## 에스컬레이션 규칙
```

#### 섹션별 작성 기준

| 섹션 | 작성 기준 |
|------|---------|
| **역할 및 책임** | 이 에이전트가 담당하는 실무 범위, 관할 팀/산출물 목록, 권한 한계 명시 |
| **트리거 조건** | 어떤 입력 또는 이벤트에서 활성화되는지 — 상위 에이전트 호출, 인계 이벤트 파일 감지, 스케줄 트리거 등 |
| **워크플로우 단계별 수행 지침** | 각 단계마다 `입력 → 처리 내용 → 출력 경로 → 성공 기준 → 실패 시 처리` 5개 항목 완전 정의 |
| **사용 스킬 목록** | 참조 스킬명, 호출 시점, 스킬 경로(`.claude/skills/{스킬명}/`) 명시 |
| **핵심 분기 확인 기준** | 사람 확인이 필요한 시점 명시 (4.4절 기준 준수) |
| **에스컬레이션 규칙** | 재시도 초과 시 보고 대상, 에스컬레이션 파일 형식, 상위 에이전트 통보 방법 |

### 4.3 스킬 트리거 규칙

**원칙**: 모든 스킬은 **워크플로우 단계 진입 시 자동 실행**. 키워드 매칭·슬래시 커맨드 불필요.

| 트리거 방식 | 적용 여부 |
|-----------|---------|
| 워크플로우 단계 자동 진입 | ✅ 기본 방식 |
| 키워드 매칭 (`"IA 만들어줘"` 감지) | ❌ 사용 금지 |
| 슬래시 커맨드 직접 호출 | ❌ 사용 금지 |

**구현 방법**: 각 AGENT.md의 `## 워크플로우 단계별 수행 지침` 에 `해당 단계 도달 시 {스킬명} 스킬 즉시 참조` 문구를 명시하여, 에이전트가 해당 단계에 진입하면 자동으로 스킬을 호출한다.

### 4.4 핵심 분기 확인 기준 (Human Confirmation Checkpoints)

다음 **3가지 상황에서만** 사람 확인을 요청한다. 그 외 중간 산출물은 자동 진행.

| # | 확인 시점 | 이유 | 확인 대상 |
|---|----------|------|---------|
| 1 | **WBS 납기일 확정** | 전체 6단계 일정과 팀별 마일스톤이 납기에 종속됨 — 잘못 확정되면 전 단계 재작성 필요 | 프로젝트 착수 시 (Step 7 직전) |
| 2 | **IA 1depth 메뉴 확정** | 1depth 구조가 확정되어야 화면설계서 전체 화면 목록이 결정됨 | Step 9 FO/BO 1depth 도출 직후 |
| 3 | **외부 발주처 명의 문서 최종 발행** | 고객사 이름이 들어간 제안서·착수보고서는 오류 발송 불가 | Step A-5 이사님 검토 요청 전 |

**구현 방법**: 각 AGENT.md에서 위 시점 도달 시 `[확인 요청]` 블록을 출력하고 진행 중단. 사용자 승인 입력 후 다음 단계 재개.

### 4.5 동시 실행 충돌 방지

**방식**: L1 비서실이 **동일 프로젝트 내 에이전트 위임을 순차 직렬화**.

#### 규칙

1. 같은 `{프로젝트명}` 을 처리하는 에이전트는 **동시에 2개 이상 실행 금지**
2. L1은 이전 에이전트의 **산출물 파일 존재 확인** 후 다음 에이전트 위임
3. 서로 다른 `{프로젝트명}` 의 에이전트는 **병렬 실행 허용**
4. 진행 상태는 `/output/{자회사}/{프로젝트명}/.status` 파일로 관리

#### 상태 파일 형식 (`.status`)

```
project: {프로젝트명}
current_step: Step 9
locked_by: planning-team
locked_at: YYYY-MM-DDTHH:MM:SS
last_output: ia-{프로젝트명}.md
```

**충돌 감지 시 처리**: `.status` 파일에 `locked_by` 가 존재하면 후행 에이전트는 대기 후 에스컬레이션 통보.

### 4.6 실패 복구 프로토콜

**원칙**: 이유 명시 → 사용자 수정 → 해당 단계 재실행 (처음부터 아님).

| 단계 | 처리 내용 |
|------|---------|
| 1. 실패 감지 | 검증 실패 또는 처리 불가 시 에이전트가 `[실패 리포트]` 생성 |
| 2. 이유 명시 | 실패 원인, 누락 항목, 재실행에 필요한 정보를 구체적으로 서술 |
| 3. 자동 재시도 | 최대 2회 자동 재시도 (동일 입력으로) |
| 4. 사용자 수정 | 재시도 2회 초과 시 사용자에게 시나리오 수정 또는 입력 보완 요청 |
| 5. 단계 재실행 | 수정 입력으로 해당 단계만 재실행 — 이전 완료 단계는 유지 |

**실패 리포트 필수 항목**:
- 실패 단계 코드 (예: `Step 9 / DE-02`)
- 실패 원인 요약 (1~2문장)
- 누락 또는 불충분한 입력 목록
- 재실행 시 필요한 추가 정보
