# site-analysis-rules.md

## 개선 포인트 작성 기준

### 우선순위 판단 기준

| 우선순위 | 조건 |
|---------|------|
| 상 | 사이트 접근성·보안 직접 영향 (SSL 오류, 모바일 미지원, 로딩 3초 초과) |
| 중 | UX 개선으로 전환율 향상 가능 (내비게이션 구조, CTA 부재, 검색 불편) |
| 하 | 부가 기능 또는 콘텐츠 관련 (SNS 연동, FAQ 보강, 블로그 정기 업데이트) |

### 유지보수 연관성 판단 기준

| 연관성 | 조건 |
|-------|------|
| 높음 | 기능 수정·버그 수정·보안 패치로 직접 해결 가능 |
| 보통 | 기획·설계 후 구현이 필요한 기능 개선 |
| 낮음 | 콘텐츠 업데이트나 디자인 리뉴얼 영역 |

### 금지 사항

- 직접 확인 불가 정보를 사실처럼 단정 짓는 표현 금지 (예: "로딩 시간이 5초이다" → "로딩 지연 가능성 있음 (측정 불가)")
- 경쟁사 비방 표현 금지
- 고객사 기술스택을 100% 확신하는 표현 금지 (반드시 "추정" 표기)
- 가격, 매출, 직원 수 등 비공개 정보 임의 기재 금지

### 정보 부족 처리

WebSearch로 충분한 정보를 수집하지 못한 경우:

1. 수집된 정보만으로 분석 가능한 섹션 작성
2. 미확인 항목에 "미확인 (공개 정보 없음)" 명시
3. 보고서 상단에 "(정보 수집 제한 — 추가 조사 권장)" 배너 추가
4. 에스컬레이션 없이 가능한 범위 내에서 진행

## 기술스택 추정 방법

직접 접근이 불가한 경우 다음 간접 방법으로 추정한다:

1. **LinkedIn/채용공고**: "React 개발자", "Node.js 백엔드" 등 키워드로 스택 추정
2. **기술 블로그/GitHub**: 공개 저장소나 기술 포스팅 확인
3. **도메인 분석**: 도메인 네임, 서브도메인 구조로 인프라 추정
4. **BuiltWith/Wappalyzer 공개 리포트**: 외부 분석 도구 결과 인용 (출처 명시)
