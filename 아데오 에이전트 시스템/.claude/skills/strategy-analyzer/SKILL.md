# strategy-analyzer

## 개요

고객사 자료 및 제안 파트 인계 문서 기반으로 아데오 표준 요구사항정의서(AY-01)를 생성한다.  
반드시 `/templates/requirements-template.md` 를 참조하여 작성한다.

---

## 실행 절차

### Step 1: 입력 확인

| 필수 입력 | 확인 방법 |
|---------|---------|
| 프로젝트명 | 사용자 입력 또는 인계 문서에서 추출 |
| 고객사 소개 자료 | 파일 경로 또는 텍스트로 제공 여부 |
| 제안 파트 인계 문서 | Google Drive 아데오 프로젝트/{프로젝트명}/.status/.status 파일의 `outputs.kickoff` URL → `mcp__claude_ai_Google_Drive__read_file_content`로 읽기 |
| 요구사항 템플릿 | `/templates/requirements-template.md` 존재 여부 |

입력 자료가 전혀 없으면 **[확인 요청]** 을 출력하고 중단.

### Step 2: 현황 분석

고객사 자료를 분석하여 다음 항목을 추출한다.

| 분석 항목 | 내용 |
|---------|------|
| 고객사 개요 | 업종, 규모, 주요 사업 |
| 현재 시스템 현황 | 기존 웹사이트/시스템 유무, 기술 스택 |
| 개선 요구사항 | 고객사가 요청한 핵심 개선 사항 |
| 참조 사이트 | 벤치마크 또는 레퍼런스 사이트 (있으면) |

### Step 3: 기능 요구사항 도출

1. 고객사 자료 + 인계 문서에서 기능 목록 추출
2. FO(Front Office) / BO(Back Office) 분리
3. 우선순위 분류: 필수(Must) / 권장(Should) / 선택(Nice-to-have)
4. 각 기능에 요구사항 코드 부여: `REQ-F-NNN` (기능), `REQ-NF-NNN` (비기능)

### Step 4: 비기능 요구사항 정의

세부 규칙은 `references/strategy-rules.md` 참조.

| 구분 | 항목 |
|------|------|
| 성능 | 응답시간, 동시접속자 수 |
| 보안 | 인증 방식, 개인정보 처리 방침 |
| 가용성 | 서비스 가동률, 장애 복구 시간 |
| 유지보수 | 코드 품질 기준, 문서화 수준 |
| 호환성 | 지원 브라우저, 반응형 기준 해상도 |
| 접근성 | WCAG 2.1 준수 여부 |

### Step 5: 제약사항 정의

| 구분 | 예시 |
|------|------|
| 기술 제약 | 고객사 지정 기술 스택, 기존 연동 시스템 |
| 일정 제약 | 납기일, 단계별 마일스톤 |
| 예산 제약 | 추가 기능 개발 범위 기준 |
| 법적 제약 | 개인정보보호법, 전자상거래법 등 |

### Step 6: 출력 및 검증

- **출력 방법**: 내용 작성 후 `mcp__claude_ai_Google_Drive__create_file`로 업로드 → 반환된 URL을 Google Drive 아데오 프로젝트/{프로젝트명}/.status/.status 파일의 `outputs.requirements`에 기록
- **검증**: `scripts/validate-doc.py` 실행
- **성공 기준**: 현황분석 섹션 + 기능 요구사항(REQ-F 5개 이상) + 비기능 요구사항 6개 구분 + 제약사항 섹션 존재

## 실패 처리

| 실패 유형 | 처리 |
|---------|------|
| 입력 자료 없음 | [확인 요청] 출력 후 중단 |
| 기능 요구사항 5개 미만 도출 | 인계 문서 재분석 1회 후 에스컬레이션 |
| validate-doc.py 실패 | 자동 재시도 1회 → 초과 시 에스컬레이션 |
